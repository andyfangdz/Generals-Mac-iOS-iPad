## 2026-07-12: iPhone on the big screen — external display + touchscreen trackpad

Plugging a USB-C monitor into an iPhone now moves the game fullscreen
onto the monitor while the phone's touchscreen becomes a relative-motion
trackpad driving the game's virtual cursor. Unplugging reverses both
mid-game. Spec and plan under `docs/superpowers/`.

Changes:
- New iOS platform layer (`iOSExternalDisplay.mm`): watches SDL's
  display hot-plug events (applied at the frame boundary, debounced
  500 ms) and migrates the SDL window's underlying `UIWindow` between
  screens. The CAMetalLayer survives the move, so DXVK experiences a
  resize rather than a surface loss. SDL3's UIKit backend only picks a
  screen at window creation, so the reassignment happens behind SDL's
  back via `SDL_PROP_WINDOW_UIKIT_WINDOW_POINTER`.
- If a monitor is attached at launch, the migration runs before the
  `-xres/-yres` injection so the game resolution derives from the
  monitor; mid-game changes reuse the options screen's resolution
  recipe (`setDisplayMode` + layout/control-bar/camera refreshes),
  split into a separate C++ TU because the engine's Bool/Byte typedefs
  collide with UIKit's MacTypes inside an ObjC++ file.
- The phone shows a minimal trackpad surface (dimmed, faint hints)
  whose UITouch handler synthesizes relative SDL mouse events with the
  indirect-pointer ID (`which = 0`): they ride the iPad pointer-lock
  virtual-cursor path in `SDL3Mouse` untouched. Gestures mirror the
  absolute translator: tap = left click (link-window double-tap =
  double click), tap-then-drag = band select, stationary long-press =
  right click, two-finger drag = camera scroll (RMB drag), pinch =
  zoom (wheel).
- The game-rendered ANI cursor now also draws when no mouse device
  exists (`SDL_HasMouse() || trackpad active`), and the absolute touch
  gesture translator is fenced off while the trackpad owns the phone
  screen (including its per-frame long-press poll, which could fire a
  phantom right-click from a finger that was mid-gesture at migration).
- `GX_EXTERNAL_DISPLAY = wired|any|off` in Options.ini. `wired`
  (default) filters AirPlay via the tv-idiom trait heuristic — iOS has
  no public wired-vs-AirPlay API, so `off` is the escape hatch and
  `any` embraces AirPlay latency.

Validation:
- Clean `z_generals` iPhoneOS arm64 builds after every task; only the
  pre-existing warning set plus the expected `UIScreen.screens`
  deprecation (the app is not scene-based, so the compatibility path
  applies).
- `enable_language(OBJCXX)` cannot initialize under the vcpkg
  toolchain this deep in the tree; the .mm compiles through the CXX
  rule with `-xobjective-c++` instead.
- On-device validation (iPhone 15+, USB-C monitor) still pending:
  hot-plug both directions, DXVK surviving the migration, trackpad
  feel/sensitivity, and the AirPlay heuristic. The window-migration
  spike is the load-bearing unknown.

Second update — the trackpad is confirmed working end-to-end on the
iPhone 17 Pro Max + USB-C monitor (cursor, clicks, drag-select; the
"dead input" during final testing turned out to be the intro cutscene,
which only ESC skips — three-finger tap now sends ESC). Getting there
surfaced two more scene-era landmines, chased with one-shot
diagnostics over several install/log-pull loops:

- The engine owns the main thread, and SDL's UIKit event pump runs the
  runloop in ~2µs slices that never reach the waiting state — so
  CoreAnimation never committed new windows (the trackpad window was
  key/visible/correct-frame yet pitch black) and UIKit's event fetcher,
  which dispatches touches via the main dispatch queue, starved (a
  dispatch_after probe never fired in 10+ s). Fixes: [CATransaction
  flush] when creating/moving windows, and one real <=1ms runloop
  service window per frame while in trackpad mode.

- The runtime resolution change calls SDL_RaiseWindow, which on iOS is
  makeKeyAndVisible on the game window — putting the app's key window
  on the NON-INTERACTIVE external scene, where iOS silently drops all
  touch input. The trackpad window is re-asserted key after the
  resolution apply.

Update, same day — first device test (iPhone 17 Pro Max) failed with a
black screen on both displays, and the session log told a wild story:
SDL3's UIKit backend is scene-based, its scene delegate invokes
SDL_main once per connecting scene, and iOS connects a second
UIWindowScene for an attached monitor. The re-entrant second main()
rotated the log mid-session, failed window creation ("Only one window
allowed per display"), and its error path's SDL_Quit() tore down video
under the running game — SDL_ShowWindow silently failed, so nothing
ever appeared. The UIWindow.screen migration was also a silent no-op
in a scene-based app (drawable stayed 956x440). Fixed by refusing
main() re-entry and migrating via UIWindow.windowScene onto the very
scene iOS connects for the monitor, with the poll reconciling until
the scene exists and the engine can take a resolution change.

## 2026-07-11: Lock the iPad pointer and restore game cursors

Replaced the native iPadOS pointer path with SDL relative mouse
mode and the game's existing W3D polygon cursor renderer.

Changes:
- iPad mouse input now uses `SDL_SetWindowRelativeMouseMode`,
  keeping the system pointer hidden and confined to the game
  instead of allowing edge movement to expose system UI.
- Relative SDL motion drives a clamped virtual cursor in game
  coordinates, while synthetic touch events remain absolute.
- Software cursor drawing follows `SDL_HasMouse()` so iPadOS
  global mouse IDs cannot incorrectly suppress the game cursor.
- Default and `NONE` cursor states render the retail
  `SCCPointer.ani` cursor because iPadOS' native fallback is hidden.
- The complete retail `Data/Cursors/*.ani` set is decoded through
  SDL_image and uploaded lazily as W3D textures, preserving cursor
  animation timing and hotspots without exposing a native cursor.
- ANI `rate` and `seq` chunks drive per-step timing and frame order;
  the cursor cache accepts the retail maximum of 33 frames instead
  of truncating at the legacy 21-frame texture limit.
- Game-rendered cursors scale by SDL's logical-to-pixel window ratio, so
  their original 32-point size remains legible on Retina displays.
- Directional cursor selection follows the W3D scroll direction,
  while context changes select the corresponding retail ANI family.
- Relative-mode clicks retain the virtual cursor position even
  when SDL reports the button event with its zero-valued global
  mouse ID, preventing offset clicks and accidental drag-selects.
- Mouse-to-touch synthesis is disabled and indirect touch devices
  are excluded from the screen gesture translator, preventing one
  trackpad click from being delivered again at touch coordinates.
- Synthetic touch-to-mouse events are explicitly tagged with
  `SDL_TOUCH_MOUSEID`, preventing touch input from being treated
  as an external mouse.
- One-finger double taps now emit an actual SDL double-click after
  two taps within 500 ms and 32 window points; a 12-point circular
  touch dead zone tolerates finger jitter without making drag-select
  difficult to begin.
- Physical iPad pointers accept up to 48 points of relative movement
  between clicks, reducing trackpad double-click failures on the
  high-DPI display while retaining SDL's 500 ms time limit.
- The shared W3D cursor path now updates polygon cursor state on
  non-Windows platforms; the SDL implementation remains unchanged
  on Linux and macOS.

Validation:
- Built the complete Zero Hour `z_generals` target for iPhoneOS
  arm64 and produced a successfully signed development app bundle.
- Rebuilt the iPhoneOS target after adding touch and trackpad
  double-click handling; compilation and linking succeeded with
  only the pre-existing warning set.
- Built an arm64 iOS Simulator engine, SDL shell, and simulator
  DXVK libraries, then installed and launched the assembled app on
  an iPad Pro 11-inch simulator running iOS 26.5.
- Simulator startup successfully loaded game assets, SDL, OpenAL,
  DXVK, and MoltenVK. Rendering could not reach the menu because
  the simulated Apple GPU rejected a Vulkan 1.2 device feature
  required by DXVK (`VK_ERROR_FEATURE_NOT_PRESENT`). This is a
  Simulator GPU limitation, so pointer-lock behavior and visual
  cursor switching still require final validation on a physical
  iPad.
